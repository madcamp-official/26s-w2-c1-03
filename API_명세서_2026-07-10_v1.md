# trip and end — Backend API 명세서 (v1)

> 기준 문서: `AI_기반_여행_계획_및_기록_앱_기능명세서_2026-07-10_v2.md`, `trip_and_end_erd.dbml`
> 이 문서는 백엔드 구현 착수 전 API 설계 초안이며, 실제 구현 중 변경될 수 있다.

---

## 0. 공통 사항

| 항목 | 내용 |
|---|---|
| Base URL | `https://api.tripandend.app/v1` (예시, 실제 도메인 확정 전) |
| 인증 | JWT — `Authorization: Bearer <accessToken>` |
| 응답 포맷 | JSON (`application/json`), 시각은 ISO 8601 (UTC) |
| 에러 포맷 | `{ "error": { "code": "STRING_CODE", "message": "..." } }` |
| 페이지네이션 | cursor 기반: 쿼리 `?cursor=&limit=`, 응답 `{ items: [...], nextCursor: string\|null }` |
| 권한 모델 | `trip_members.role` (`owner` / `editor` / `viewer`) 기준으로 여행 관련 리소스 접근 제어 |

### JWT 정책
- Access Token: 만료 30분, 클레임에 `userId` 포함
- Refresh Token: 만료 30일, DB(서버)에 해시로 저장, 재발급 시 회전(rotation) + 재사용 탐지 시 해당 유저의 전체 세션 무효화
- 클라이언트 저장: `flutter_secure_storage` (Keychain/Keystore), 평문 저장 금지

### WebSocket 인증
- 연결 시 `wss://api.tripandend.app/v1/ws/trips/{tripId}?token={accessToken}`
- 서버는 연결 수립 시 토큰 검증 + `trip_members`에 해당 유저가 소속되어 있는지 확인, 미소속/토큰 만료 시 4403으로 close

## 2. 여행 계획 (Trip Plan)

### 2.1 여행 생성/조회/수정/삭제

| Method | Endpoint | 설명 | 요청 | 응답 |
|---|---|---|---|---|
| POST | `/trips` | 도시/날짜 입력으로 여행 생성 (`status=planning`, 생성자는 `trip_members.role=owner`로 자동 등록) | `{ title, cityName, areaCode, sigunguCode, startDate, endDate }` | `trip` |
| GET | `/trips` | 내가 속한 여행 목록 (`?status=planning\|ongoing\|completed`) | - | `{ items: trip[], nextCursor }` |
| GET | `/trips/{tripId}` | 여행 상세 (요약 정보) | - | `trip` |
| PATCH | `/trips/{tripId}` | 제목/기간 등 수정 (`owner`\|`editor`) | `{ title?, startDate?, endDate? }` | `trip` |
| DELETE | `/trips/{tripId}` | 여행 삭제 (`owner`만, soft delete) | - | `204` |

### 2.2 관광지 후보 추천

사용자 흐름: **도시/날짜 입력 → 후보 관광지 목록 제시(인기순) → 카테고리 선택 시 지도에 마커로 필터링(마커 클릭으로 선택 가능) → 사용자가 마음에 드는 곳 선택 → AI가 선택된 장소로 최적 동선 스케줄 생성**

| Method | Endpoint | 설명 | 요청 | 응답 |
|---|---|---|---|---|
| GET | `/trips/{tripId}/places/candidates` | `area_code`/`sigungu_code` 기준 TourAPI 장소 조회 → `places`에 캐싱 → 각 장소의 Kakao 리뷰수/평점을 병합 조회해 **인기순(평점·리뷰수 가중치 정렬)**으로 정렬한 후보 목록 반환 | `?category=` (선택, 서버 사이드 사전 필터용) | `{ candidates: place[] }` |
| GET | `/places/{placeId}` | 장소 상세 (캐시된 `places` row) | - | `place` |

`place` 스키마: `{ id, source, name, address, lat, lng, categoryCode, imageUrl, overview, tel, rating, reviewCount }`

> **지도 마커 표시**: 카테고리 선택 시 마커 필터링과 마커 클릭을 통한 장소 선택은 위 후보 목록(좌표 포함) 안에서 클라이언트가 처리한다. 후보 범위를 벗어나 도시 전체 장소를 별도로 재조회하지 않는다.
> **인기순 산정**: TourAPI 목록을 가져온 뒤 Kakao 로컬 API에서 각 장소의 평점/리뷰수를 조회해 병합 정렬한다. Kakao에 매칭되지 않는 장소는 최하위로 정렬한다(정확한 매칭 로직은 구현 단계에서 상세화 필요 — §8 참고).

### 2.3 후보 선택 → AI 스케줄 생성

| Method | Endpoint | 설명 | 요청 | 응답 |
|---|---|---|---|---|
| POST | `/trips/{tripId}/schedule/generate` | 사용자가 선택한 장소들로 AI가 일자별 최적 동선 생성. 각 선택 장소는 사용자가 고른 날짜(`dayNumber`)에 강제 배정되고, `trip_places`에 `day_number`/`order_in_day` 배정하여 bulk insert (`added_by`=요청자). **동기 처리** — 응답이 올 때까지 클라이언트는 로딩 화면을 띄운다 | `{ selectedPlaces: { placeId: string; dayNumber: number }[] }` | `{ schedule }` |

> v1은 동기 방식으로 구현한다(비동기 잡 큐/폴링 엔드포인트는 도입하지 않음). 완료 시 REST 응답과 별개로 WebSocket `schedule:generated` 이벤트도 트립 채널에 브로드캐스트해 다른 협업 멤버 화면을 갱신한다. 실사용에서 OpenAI 응답 지연이 문제가 되면 이후 버전에서 비동기(202+폴링/WS 알림)로 전환을 검토한다.

### 2.4 스케줄 조회 및 수동 편집

| Method | Endpoint | 설명 | 요청 | 응답 |
|---|---|---|---|---|
| GET | `/trips/{tripId}/schedule` | `trip_places`를 `day_number`, `order_in_day` 순으로 조회 | - | `{ days: [{ dayNumber, places: tripPlace[] }] }` |
| POST | `/trips/{tripId}/schedule/places` | 장소 수동 추가 (`place_id` 참조 또는 `custom_name`/`custom_address` 직접입력) | `{ placeId? , customName?, customAddress?, dayNumber, orderInDay?, memo? }` | `tripPlace` |
| PATCH | `/trips/{tripId}/schedule/places/{tripPlaceId}` | 메모 수정, 개별 위치 이동 | `{ dayNumber?, orderInDay?, memo? }` | `tripPlace` |
| DELETE | `/trips/{tripId}/schedule/places/{tripPlaceId}` | 장소 제거 | - | `204` |
| PATCH | `/trips/{tripId}/schedule/reorder` | 드래그앤드롭 등 일괄 순서 변경 | `{ operations: [{ tripPlaceId, dayNumber, orderInDay }] }` | `{ days: [...] }` |

### 2.5 프롬프트 기반 AI 재수정

| Method | Endpoint | 설명 | 요청 | 응답 |
|---|---|---|---|---|
| POST | `/trips/{tripId}/schedule/revise` | 사용자가 자연어 프롬프트로 계획 전체 재수정 요청. `ai_plan_requests`에 `prompt_text` 기록, AI 응답 후 `response_summary` 저장 및 `trip_places` 갱신 | `{ prompt }` | `{ schedule, requestId }` |
| GET | `/trips/{tripId}/ai-requests` | AI 계획 생성/수정 요청 이력 조회 | - | `{ items: aiPlanRequest[] }` |

### 2.6 여행 대표사진

| Method | Endpoint | 설명 | 요청 | 응답 |
|---|---|---|---|---|
| PUT | `/trips/{tripId}/cover` | 여행 목록에 표시될 대표사진 지정 | `{ recordPhotoId }` | `trip` |
| DELETE | `/trips/{tripId}/cover` | 대표사진 해제 (기본 플레이스홀더 이미지로 표시) | - | `trip` |

**대표사진 규칙** (§4 기록 비공개 원칙과 연동)
- 기록은 작성자 본인에게만 비공개이므로, `recordPhotoId`는 **요청자 본인이 작성한 기록의 `record_photos`**여야 한다(타 멤버 사진 지정 시 `403`).
- 트립 멤버 누구나(자기 사진에 한해) 대표사진으로 지정 가능. 먼저 지정한 사진이 유지되며, 이후 다른 멤버가 재지정하면 덮어쓴다.
- 대표사진으로 지정된 `record_photo`가 삭제되면(개별 삭제 또는 기록 전체 삭제, §4·§5) 서버가 `trips.cover_image_url`을 자동으로 `null` 처리하고, 클라이언트는 기본 플레이스홀더 이미지를 표시한다.
- 이후 아무 멤버나(자기 사진 중에서) 다시 대표사진을 지정할 수 있다.

## 3. 공동 여행 계획 수립 (Collaboration)

### 3.1 초대 / 멤버 관리 (REST)

| Method | Endpoint | 설명 | 요청 | 응답 |
|---|---|---|---|---|
| POST | `/trips/{tripId}/invite-links` | 공유 링크 생성 (`owner`\|`editor`) | `{ expiresInHours? }` | `{ token, url, expiresAt }` |
| POST | `/trips/invite-links/{token}/join` | 초대 링크로 참여 (로그인 필요, `trip_members` insert, 기본 `role=editor`) | - | `{ tripId }` |
| GET | `/trips/{tripId}/members` | 참여자 목록 | - | `{ items: member[] }` |
| PATCH | `/trips/{tripId}/members/{userId}` | 역할 변경 (`owner`만) | `{ role }` | `member` |
| DELETE | `/trips/{tripId}/members/{userId}` | 멤버 내보내기 (`owner`만) | - | `204` |
| DELETE | `/trips/{tripId}/members/me` | 자진 탈퇴 | - | `204` |

### 3.2 실시간 동기화 (WebSocket)

- 채널: `wss://.../v1/ws/trips/{tripId}?token={accessToken}`
- 연결 시 서버가 `trip_members` 소속 확인 후 accept, 이후 REST로 변경된 스케줄도 동일 채널에 broadcast

**클라이언트 → 서버**

| 이벤트 | payload | 설명 |
|---|---|---|
| `schedule:op` | `{ opId, type: "add"\|"remove"\|"move"\|"editMemo", tripPlaceId?, dayNumber?, orderInDay?, memo? }` | 편집 동작 전파 |
| `presence:ping` | `{}` | 접속 유지/온라인 상태 표시 |

**서버 → 클라이언트**

| 이벤트 | payload | 설명 |
|---|---|---|
| `schedule:op` | 위와 동일 + `authorUserId` | 다른 멤버의 편집을 나머지 참여자에게 브로드캐스트 |
| `schedule:generated` | `{ schedule }` | AI 스케줄 생성 완료 알림 (2.3 비동기 응답) |
| `schedule:conflict` | `{ tripPlaceId, serverState }` | 낙관적 잠금 충돌 시 서버 최신 상태 강제 전달 |
| `member:joined` / `member:left` | `{ userId, nickname }` | 참여자 입퇴장 |

**충돌 처리 원칙**: `trip_places` 각 행은 `updated_at` 기반 낙관적 잠금(버전 비교)을 사용한다. 동시 편집 시 먼저 도착한 변경을 반영하고, 이후 도착한 변경이 stale하면 거부 후 `schedule:conflict`로 최신 상태를 클라이언트에 강제 동기화한다. CRDT/OT 등 정교한 병합은 1차 구현 범위 밖으로 두고 향후 과제로 남긴다.

## 4. AI 기반 여행 사진 선별 및 기록

> 전제: `trips.status`는 `end_date` 다음날 서버 배치가 `completed`로 전환하며, 이때 `notification_logs(type=trip_end_reminder)`를 남기고 등록된 `user_devices`로 푸시를 보낸다. 사진첩 조회는 사용자가 알림 클릭 또는 앱 내 "기록 시작"을 누른 시점에만 시작된다.
>
> **비공개 원칙**: `travel_records`/`record_photos`는 **작성자 본인만** 조회·수정·삭제할 수 있다. 같은 트립의 다른 멤버라도 조회 권한이 없으며, 아래 모든 엔드포인트는 `record.user_id == 요청자` 검증 후 아니면 `403`을 반환한다.

| Method | Endpoint | 설명 | 요청 | 응답 |
|---|---|---|---|---|
| POST | `/trips/{tripId}/records` | 기록 세션 시작. `travel_records` insert(`status=draft`). `(trip_id, user_id)` unique이므로 기존 draft가 있으면 그대로 반환 | - | `record` |
| POST | `/trips/{tripId}/records/{recordId}/photos/metadata` | 온디바이스 1차 필터(흔들림/노출/중복 제거, OCR 문서 제외, 얼굴 감지 감점) 통과 사진의 텍스트 메타데이터만 배치 등록 (`takenAt`, `locationName`, `localId`). 최종 선택 전이라 `record_photos`에는 미기록, 서버가 임시 참조 ID 발급 | `{ photos: [{ localId, takenAt, locationName }] }` | `{ photos: [{ photoRefId, localId }] }` |
| POST | `/trips/{tripId}/records/{recordId}/photos/upload` | 1차 필터 통과 사진 실물 업로드 (앱→백엔드 직접, multipart, EXIF 완전 제거 상태, 최대 100장). 임시 버퍼(pass-through, TTL 부여)에만 저장, 디스크/DB 미기록 | `multipart/form-data`: `photoRefId`별 파일 | `{ uploaded: photoRefId[] }` |
| POST | `/trips/{tripId}/records/{recordId}/photos/curate` | `takenAt` 기준 일자별 그룹핑 후 OpenAI 배치 전송, 여행 일수 기준 동적 배분으로 최종 최대 15장 추천. 비추천분은 즉시 임시 버퍼에서 폐기 | - | `{ recommended: photoRefId[] }` (최대 15개) |
| GET | `/trips/{tripId}/records/{recordId}/photos/candidates` | 추천된 사진의 임시 미리보기(짧은 TTL 서명 URL) 조회 | - | `{ items: [{ photoRefId, previewUrl, takenAt, locationName }] }` |
| POST | `/trips/{tripId}/records/{recordId}/photos/finalize` | 사용자 최종 선택 확정. 선택된 사진만 암호화 스토리지에 영구 업로드 + `record_photos` insert. 나머지는 임시 버퍼에서 전량 폐기 | `{ selections: [{ photoRefId, caption?, orderIndex? }] }` | `{ recordPhotos: recordPhoto[] }` |
| PATCH | `/trips/{tripId}/records/{recordId}/photos/{recordPhotoId}` | 캡션/순서/대표사진(`is_cover`) 수정 | `{ caption?, orderIndex?, isCover? }` | `recordPhoto` |
| DELETE | `/trips/{tripId}/records/{recordId}/photos/{recordPhotoId}` | 개별 사진 삭제 (스토리지 파일도 함께 삭제, 대표사진으로 지정된 사진이면 `trips.cover_image_url` 자동 해제 — §2.6) | - | `204` |
| GET | `/trips/{tripId}/records/{recordId}/writing-template` | AI 글쓰기 템플릿 제안 (선택 기능) | - | `{ template: string }` |
| PATCH | `/trips/{tripId}/records/{recordId}` | 일기 본문 작성/수정, `draft`→`published` 전환 | `{ title?, content?, status? }` | `record` |

**임시 버퍼 폐기 흐름 정리**
1. 업로드 시점: 최대 100장 전량 임시 버퍼(pass-through, DB/디스크 미기록)에 존재
2. `curate` 완료 시점: OpenAI가 추천하지 않은 사진(최대 ~85장)은 즉시 폐기
3. `finalize` 완료 시점: 사용자가 선택하지 않은 추천 사진(최대 15장 중 미선택분)도 전량 폐기, 선택분만 암호화 스토리지로 이관

|---|---|---|---|---|
| GET | `/records` | 내 모든 여행 기록 목록 — **본인이 작성한 기록만** 반환(여행 기간, `cityName`, 대표사진 요약) | `?cursor=&limit=` | `{ items: recordSummary[], nextCursor }` |
| GET | `/records/{recordId}` | 기록 상세 (사진 목록 포함, 작성자 본인만) | - | `record` |
| DELETE | `/records/{recordId}` | 기록 삭제. `travel_records.deleted_at` soft delete + 연결된 `record_photos`는 스토리지 파일까지 완전 삭제(hard delete). 삭제된 사진 중 트립 대표사진이 있었다면 `trips.cover_image_url` 자동 해제(§2.6) | - | `204` |

## 6. 보안 / 구현 원칙 요약

| 구분 | 원칙 |
|---|---|
| 인증 | JWT access(30분)/refresh(30일, rotation + 재사용 탐지), 클라이언트는 `flutter_secure_storage` 사용 |
| WebSocket | 연결 시 JWT 검증 + `trip_members` 소속 확인, 미소속 시 close |
| 전송 구간 | 앱↔백엔드 TLS 1.2+, 백엔드↔OpenAI TLS(기본 제공), API 키는 Secret Manager 관리 |
| 저장 암호화 | 최종 저장 사진(`record_photos.storage_url`) 및 텍스트 메타데이터 모두 Encryption at Rest 적용 |
| 임시 버퍼 | **로컬 서버 디스크의 임시 디렉터리**에 pass-through 저장(단일 서버 배포 전제) + 강제 TTL(예 30분) cron 삭제 이중 안전장치. `curate`/`finalize` 완료 시 명시적 삭제도 병행. 다중 서버로 수평 확장 시 공유 스토리지 방식 재검토 필요 |
| OpenAI 전송 | 서버가 전송 직전 EXIF 재검증(이중 스트립) 후 전송, 원본 좌표·EXIF·이미지 바이너리는 로그에 남기지 않음 |
| 사진 업로드 | 앱 → 백엔드 직접 업로드(멀티파트), 서버는 스트리밍 처리로 원본을 디스크에 평문으로 남기지 않고 암호화 저장 |
| 삭제 정책 | 여행/기록은 soft delete, 사진 실물은 삭제 요청 시 스토리지에서 hard delete |
| 로깅 | 사진 바이트, 원본 GPS 좌표, 소셜 로그인 토큰 원문은 어떤 로그에도 남기지 않음 |
